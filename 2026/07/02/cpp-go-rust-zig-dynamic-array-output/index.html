4. Rust：Vec 的 Debug 和 Display 区别

Rust 严格区分两类格式化：

  • Debug：给开发者调试使用，对应 {:?} 或 {:#?}
  • Display：给用户展示使用，对应 {}

Vec<T> 实现了 Debug，但没有实现 Display。所以 println!("{:?}", v) 合法，println!("{}", v) 会编译报错。

底层原因对比

C 的设计非常接近机器模型。数组名在表达式中经常退化为指针，标准库也没有数组长度信息，所以语言和库都无法自动知道“这个数组应该打印几个元素”。

C++ 的核心目标是零开销抽象。早期标准库没有默认给所有容器提供流输出能力，避免把展示格式强加给所有工程。C++23 标准化范围格式化之后，标准库层面开始提供更统一的容器输出方式。

Go 更偏向工程实用性。fmt 可以结合运行时类型信息输出常见容器，调试时非常省事。

Rust 把调试输出和用户展示明确拆开。Debug 适合开发阶段观察数据结构，Display 则要求开发者给出稳定、面向业务的展示格式。

Zig 的思路更接近显式控制。ArrayList 本身不是直接打印对象，而是通过 .items 拿到底层切片，再交给格式化器输出。

工程建议

日常调试时，Go 可以直接 fmt.Println(v)，Rust 固定使用 {:?}，Zig 打印 list.items，C 手写循环，旧版 C++ 可以准备一个小的输出辅助函数。

生产输出时，不要把调试格式直接当成业务协议。尤其是 Rust 的 Debug 输出和 C++ 自定义 operator<<，都应该确认格式是否稳定、是否适合外部依赖。

跨语言学习时，可以把这件事当成语言设计入口：同样是打印 [1, 2, 3]，背后其实牵涉到类型系统、标准库边界、运行时信息、格式化接口和工程哲学。

总结

动态数组直接输出看起来只是一个调试小问题，但它背后反映了不同语言对“便利性、显式性、标准库职责和工程稳定性”的取舍。

如果只看调试便利性，Go 和 Zig 最直接；如果看底层控制，C 和 C++ 更需要开发者自己处理展示逻辑；如果看格式语义，Rust 的 Debug / Display 分层最清晰。
