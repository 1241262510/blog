GLM 5.1 开放之后，我先注意到的是成本

摘要

GLM 5.1 终于向 Code Plan 订阅者开放了。对一直在等配套服务的人来说，这次补齐本身就值得关注；但真正让我快速建立印象的，不是能力宣传，而是实际使用中非常直接的成本感知。

正文

GLM 5.1 来了

对 Code Plan 订阅者来说，这次更新算是一件实打实的好事。

之前因为算力或者套餐策略的原因，GLM 5.x 的一些配套能力始终没有完整落到 Code Plan 这条线上。既然同样是付费用户，却总要比别人慢一拍，这种体验说不上公平，也很难让人满意。

从用户角度看，花钱多少可以影响配额和速度，但如果长期连核心能力都无法及时覆盖，落差感会很明显。

不过这次情况总算变了。官方终于把对应服务补上，Code Plan 的订阅者也能正式用上 GLM 5.1。

这张图对应的是官方侧能力开放后的实际界面信息，对 Code Plan 用户来说，关键点不是“模型发布了”，而是“终于能用了”。

对主流 Agent App 的支持也跟上了

从现有信息看，GLM 5.1 对主流的 Agent 类应用也已经有了比较完整的支持。

这类支持的意义并不只是“能不能接入”，而是它决定了模型能不能真正进入日常工作流。只有接进你平时正在使用的工具链里，这类更新才算真正有价值。

这次我用 OpenCode 做了简单体验。整体接入过程不复杂，至少从“能不能开始用”这个层面来说，门槛已经不高了。

先看接入入口，主流 Agent 工具链已经开始给出对应支持。

再往下是模型选择和接入配置，整体思路已经比较标准化。

这一步更接近日常使用场景：不是单纯“能不能配”，而是“能不能顺手接进现有工作流”。

从实际界面来看，官方这次对常见 Agent App 的兼容度已经不像早期那样零散了。

至少在演示层面，OpenCode 这类工具已经可以把 GLM 5.1 放进正常使用链路里。

第一感受：贵，真的贵

我目前使用时间不长，所以还谈不上对模型能力有多深入的判断。

但有一点感受非常直接：token 消耗确实让我吃了一惊。

有些模型在宣传期看起来很强，真正放进 Agent 工作流里跑起来以后，才会意识到什么叫“花钱如流水”。尤其是当上下文变长、调用次数变多、任务开始反复试错时，成本会迅速堆起来。

这也是为什么很多时候，模型能力不能只看“上限”，还要看它是否适合进入你的日常使用场景。

总结

如果只说结论，我现在的看法是：

  • GLM 5.1 对 Code Plan 用户开放，本身是一个值得欢迎的补齐。
  • 真正跑起来之后，成本感知比我预期得更强。
  • 这类模型更适合用在复杂任务、长链路任务和真正需要它能力上限的场景里。

至于那些非常轻量的需求，比如：

  • 查天气
  • 写一点简单脚本
  • 做基础问答
  • 处理低复杂度编程任务

未必需要把最贵的模型顶上去。

更现实的做法可能还是那句老话：好钢用在刀刃上。

把高成本模型留给真正复杂、真正有价值的任务，其他场景让更便宜的大模型去接手，整体体验和成本控制都会更合理一些。
